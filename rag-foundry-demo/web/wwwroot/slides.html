What we'll build in the next 30 minutes

The product

A console assistant that answers questions about a mining operations knowledge base — safety, equipment, shift procedures.

Every answer is grounded in the docs and shows its citations.

The outcome

  • ingest → builds a searchable vector index
  • chat → grounded Q&A with sources
  • Asks something off-topic → it says "I don't know" instead of making it up
Frame the whole demo here. The "I don't know" moment is the punchline — it's the difference between a grounded assistant and a confident liar.

The problem with a raw LLM

Without grounding

  • Doesn't know your data — policies, manuals, runbooks
  • Frozen at its training cutoff
  • Hallucinates confidently when it doesn't know
  • No source you can verify or audit

With RAG

  • Answers from your private content
  • Update the docs → update the answers, no retraining
  • Grounded → far fewer hallucinations
  • Returns citations for trust & compliance

RAG = give the model the right facts at question time, then ask it to answer using only those facts.

RAG in one slide

1 · Retrieve  →  2 · Augment  →  3 · Generate

1 · Retrieve

Embed the question, search the index, get the most relevant chunks.

2 · Augment

Stuff those chunks into the prompt as grounding context.

3 · Generate

Model answers using only that context, with citations.

"Conceptually simple — the hard parts are retrieval quality and prompt discipline." We'll see both.

Emphasise: the LLM is the easy part. 80% of RAG quality is the retrieval pipeline — chunking, embeddings, hybrid search, ranking.

Why Azure AI Foundry

One platform for the pieces

  • Model catalog + managed deployments (chat & embeddings)
  • Azure AI Search as the recommended vector store
  • Enterprise security: Entra ID / RBAC, private networking, content safety

Two models we deploy today

  • gpt-4.1 — fast, low-latency Q&A (Foundry's pick for real-time RAG)
  • text-embedding-3-small — turns text into 1536-dim vectors

Swap in gpt-5-mini for reasoning, or -3-large for higher-quality recall.

Anatomy of one chat turn

sequenceDiagram autonumber actor U as User participant A as App (RagChatService) participant L as gpt-4.1 participant E as Embeddings participant S as Azure AI Search U->>A: question + chat history A->>L: condense follow-up → standalone query L-->>A: "What gloves are required on the haul road?" A->>E: embed(standalone query) E-->>A: query vector A->>S: hybrid + semantic search S-->>A: top-K grounded chunks A->>L: system rules + history + sources + question L-->>A: grounded answer + [citations] A-->>U: answer + source list

Step 2 (the condense) is what makes multi-turn work: "and what about gloves?" is rewritten into a standalone query before retrieval.

This is the per-question loop. The condense step (2-3) resolves pronouns and ellipsis against history so retrieval gets a self-contained query. History is also replayed into the final generate call (step 9) so the answer is conversational.

Step 1 — Ingest: chunk → embed → index

// 1. Read docs and split into overlapping, citable chunks
var chunks = DocumentChunker.LoadAndChunk("data");   // ~500-token windows

// 2. Turn each chunk into a vector with the Foundry embedding model
var vectors = await embedder.EmbedBatchAsync(chunks.Select(c => c.Text));

// 3. Push text + vector into Azure AI Search
await indexManager.UploadAsync(chunks, vectors);
Console.WriteLine($"Indexed {chunks.Count} chunks.");

Chunking matters: too big → noisy context & wasted tokens; too small → lost meaning. Overlap preserves context across boundaries.

The index: vector + text + semantic

var fields = new List<SearchField>
{
    new SimpleField("id", SearchFieldDataType.String) { IsKey = true },
    new SearchableField("title")  { IsFilterable = true },
    new SearchableField("source") { IsFilterable = true },   // for citations
    new SearchableField("chunk"),                            // keyword search
    new VectorSearchField("contentVector", 1536, "vector-profile") // similarity
};

var vectorSearch = new VectorSearch();
vectorSearch.Algorithms.Add(new HnswAlgorithmConfiguration("hnsw"));
vectorSearch.Profiles.Add(new VectorSearchProfile("vector-profile", "hnsw"));

var semantic = new SemanticSearch();      // re-ranks results by meaning
semantic.Configurations.Add(new SemanticConfiguration("semantic-config",
    new SemanticPrioritizedFields { TitleField = new SemanticField("title"),
        ContentFields = { new SemanticField("chunk") } }));

One index stores both the human-readable text and its vector — that's what makes hybrid search possible.

Step 2 — Retrieve: hybrid + semantic

// Embed the user's question with the SAME model used for indexing
float[] queryVector = await embedder.EmbedAsync(question);

var options = new SearchOptions
{
    Size = TopK,
    VectorSearch = new()
    {
        Queries = { new VectorizedQuery(queryVector)
                     { KNearestNeighborsCount = TopK, Fields = { "contentVector" } } }
    },
    QueryType = SearchQueryType.Semantic,                        // re-rank
    SemanticSearch = new() { SemanticConfigurationName = "semantic-config" }
};
// Passing the text + the vector = HYBRID search (keyword ∪ similarity)
var results = await searchClient.SearchAsync<KnowledgeChunk>(question, options);

Vector catches meaning ("PPE" ≈ "protective equipment"). Keyword catches exact terms & codes. Semantic ranker reorders by relevance.

Step 3 — Generate: grounded & cited

var system = """
  You are a mining operations assistant.
  Answer ONLY from the numbered SOURCES below.
  Cite sources as [1], [2]. If the answer isn't in the sources,
  say: "I don't know — that isn't in the knowledge base."
  """;

var context = string.Join("\n", results.Select((r, i) =>
    $"[{i + 1}] ({r.Source}) {r.Chunk}"));

var answer = await chatClient.CompleteChatAsync(
    [ new SystemChatMessage(system),
      new UserChatMessage($"SOURCES:\n{context}\n\nQUESTION: {question}") ]);

The system prompt is the guardrail: "only from sources" + "cite" + "say I don't know". This is where grounding is enforced.

Teach it something new — live

Update the docs → update the answers. No retraining, no redeploy.

  1. Ask: "What should I do to manage heat stress on site?" → "I don't know"
  2. Add a doc: Copy-Item demo\heat-stress-management.md data\
  3. Re-ingest: dotnet run -- ingest  (~seconds)
  4. Ask again: same question → grounded answer + [heat-stress-management.md]

The model is unchanged — we only gave it new retrievable knowledge. The same trick updates or corrects any policy, and both the local and deployed apps pick it up instantly (shared index).

This is the most memorable beat — do it live. The heat-stress doc is staged in demo/ (not indexed). Ask the question (I don't know), copy the doc into data/, run ingest, ask again. Emphasise: no fine-tuning, no model change — RAG means knowledge is just data you can edit.

What makes it production-grade

Trust & security

  • Keyless auth (Entra ID + RBAC), no keys in code
  • Citations → verifiable, auditable answers
  • Treat retrieved text as untrusted (prompt-injection)
  • Document-level security trimming in Search

Cost & latency

  • Retrieval adds round-trips + query embedding
  • Retrieved chunks add input tokens — tune TopK
  • Pick the model for the job (4.1 fast vs 5 reasoning)
  • Evaluate retrieval quality before rollout

Where to take it next

  • Foundry Agent + Azure AI Search tool — let the agent do retrieval, add memory & multi-turn
  • Integrated vectorization — Search chunks & embeds for you (skillsets/indexers)
  • Agentic retrieval — query planning, sub-queries, answer synthesis (Foundry IQ)
  • Web UI — Blazor / minimal API chat over the same services
  • Evaluation — groundedness & relevance metrics

Cost & cleanup

  • Pay-per-token for chat + embeddings (tiny for a demo corpus)
  • Azure AI Search Basic is the main standing cost — hourly while it exists
  • Tear it all down in one line:
az group delete --name rg-rag-foundry-demo --yes --no-wait

Delete the resource group after the demo so the Search service stops billing.

Key takeaways

  • RAG = Retrieve → Augment → Generate; the LLM is the easy part
  • Foundry gives you the models, AI Search, and security in one place
  • Hybrid + semantic retrieval is what makes answers good
  • The system prompt enforces grounding & citations
  • "I don't know" is a feature, not a bug
